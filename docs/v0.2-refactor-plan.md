# codexsdk-go v0.2 Refactor Plan

Status: implemented, standalone/integration verified, and released as v0.2.0

Contract revision: `codexsdk-v0.2-r2`

## Purpose

`codexsdk-go` exists only to provide Go interfaces for interacting with Codex.
Its handwritten API owns process transport, client lifecycle, typed server
request/notification delivery, and transparent composition of Codex protocol
operations.

It does not own provider-neutral LLM abstractions, business validation,
workflow, retry policy, or named application safety profiles.

## Generated Boundary

The generated `protocolv2` declarations, method registry, and `SDKSurface`
facades are protocol facts, not handwritten API design targets. Do not manually
restate or edit their method inventory.

The handwritten API must reference generated types directly and must not copy a
smaller enum, request, result, item, usage, or notification model. Generator
coverage, schema fidelity, and reproducibility remain required correctness
gates.

## Verified Current Facts

Current handwritten API includes:

- `Client`, process startup, generated facade access, and `Close`.
- `ThreadClient` with partial handwritten request/default/result models.
- A narrowed handwritten `ThreadEvent` projection.
- A handwritten server-request union and response model.
- Generated `protocolv2.ThreadStartParams`, `ThreadResumeParams`,
  `TurnStartParams`, `ThreadStartResponse`, `ThreadResumeResponse`, `Turn`,
  `ThreadTokenUsage`, `ServerNotification`, and `ServerRequest`.

The listed generated types and every response type referenced below were
verified in `codexsdk/protocolv2/protocol_types.gen.go`. Implementation must
recheck them after any protocol sync.

## Target Handwritten Public API

This section is normative. Generated declarations are intentionally omitted.

### Lifecycle Contract Shared With llmcaller-codex-go

the caller repository. The two blocks must remain byte-identical.

`ThreadRunner` adds only these semantics:

1. Call exact `thread/start` or `thread/resume` params.
2. Put the returned thread ID into exact `turn/start` params.
3. Drain or stream that turn.

`Turn.ThreadID` is composition-owned and must be empty on input; a non-empty
value fails before any request. Every other generated field passes through
unchanged. Direct fork and individual thread/turn methods remain available from
generated facades; there is no handwritten fork wrapper.

`Start`/`Resume` drain the corresponding stream implementation. Once a start or
resume response exists, subsequent failures return a partial result containing
that exact response and every observed run fact. Streaming methods report
post-start failures through `Stream.Err`, while `Stream.Result` exposes the
partial snapshot. Pre-start validation/transport failure returns a nil stream
and direct error.

`Notifications` is an ordered immutable snapshot of all exact notifications
attributable to the run. `Usage` is the last exact usage state; nil means no
usage notification was observed. `Turn` is authoritative for ID, status, error,
items, and timing. Start/resume response is authoritative for initial effective
configuration. Exact reroute notifications are authoritative for later model
changes.

`FinalResponse`, `InputStats`, and `DiagnosticRef` are allowed handwritten facts
because this lifecycle helper produces them. `FinalResponse` is a documented
projection from exact turn items and never replaces `Turn`.

### Client And Handlers

```go
package codexsdk

type ClientOptions struct {
	CWD                       string
	Command                   []string
	Initialize                protocolv2.InitializeParams
	ServerRequestHandler      ServerRequestHandler
	ServerNotificationHandler ServerNotificationHandler
	NotificationQueueCapacity int
}

type Client interface {
	SDKSurface
	ThreadRunner() ThreadRunner
	Close() error
}

func New(ClientOptions) (Client, error)

type ServerNotificationHandler func(
	context.Context,
	protocolv2.ServerNotification,
) error

type ServerRequestHandler func(
	context.Context,
	protocolv2.ServerRequest,
) (ServerRequestResponse, error)

type ServerRequestResponse struct { /* opaque tagged union */
}
```

`Initialize` is the exact generated initialization request. `New` validates
required process/initialization facts, clones mutable options, starts Codex,
initializes it, and returns an owned client.

The notification handler receives every validated generated notification on a
dedicated serial dispatcher, never on the JSON-RPC read loop. Ordering is
arrival order. The queue is bounded. Overflow, callback error, or callback panic
becomes a typed client failure; notifications are never silently dropped.

Notification ingestion has one fixed order:

1. Decode and validate the exact generated notification.
2. For a run-attributable notification, append it to the immutable run history
   and make it available to the run stream.
3. Enqueue the same value for the global user handler.

Therefore handler failure or queue overflow cannot erase a Codex fact already
accepted for a run. Global handler delivery remains ordered, but there is no
cross-goroutine happens-before promise beyond the ingestion order above.

#### Normal Close

`Close` is an orderly, synchronous shutdown:

1. Atomically enter closing state; reject new operations and stop accepting new
   transport notifications.
2. Finish active streams with `ErrClientClosed`, preserving their partial result
   and every notification accepted before the transition.
3. Deliver the already accepted global callback queue in order.
4. After the queue drains, cancel handler/client contexts, join the dispatcher,
   release transport/process resources, and return.

A handler must return for normal `Close` to complete. If it blocks permanently,
`Close` also blocks; `Close()` has no deadline contract.
The SDK does not use goroutine identity tricks to detect reentrancy. A
notification handler must not call `Close`; to request shutdown it returns an
error. Calling `Close` from the handler violates the callback contract and may
block. No `ErrReentrantClose` API is introduced.

#### Failure Shutdown

Callback error, callback panic, queue overflow, protocol failure, or transport
failure uses fail-fast shutdown instead of normal drain:

1. Record the first typed failure; later failures do not replace its cause.
2. Reject new operations and notifications.
3. Cancel the running handler context and every active stream immediately.
4. Discard callback entries still waiting in the global queue; failure shutdown
   makes no callback-drain guarantee.
5. Finish every active stream with the same first cause and its partial result.

The handler is client-scoped, so handler failure terminates the whole client,
not only one thread. A subsequent `Close` joins shutdown and returns the first
failure. It can still wait indefinitely if user handler code ignores context
cancellation. Already recorded per-run notifications remain observable even
when their global callback delivery is discarded.

`ServerRequestResponse` is necessary because the protocol has one generated
request union but no single callback response union. Its representation is
opaque and it has exactly these typed constructors:

The request-kind/response-constructor matrix is generated or mechanically
coverage-checked. Unknown future requests fail closed instead of being coerced
to an existing response.

### Errors

```go
package codexsdk

var (
	ErrClientClosed             error
	ErrStreamClosed             error
	ErrTurnFailed               error
	ErrTurnInterrupted          error
	ErrNotificationBackpressure error
	ErrHandlerFailed            error
)

type ProtocolError struct {
	RequestID protocolv2.RequestId
	Method    string
	Code      int64
	Message   string
	Data      *protocolv2.JSONValue
	Err       error
}

func (e *ProtocolError) Error() string
func (e *ProtocolError) Unwrap() error

type TurnError struct {
	ThreadID string
	Turn     protocolv2.Turn
	Err      error
}

func (e *TurnError) Error() string
func (e *TurnError) Unwrap() error
```

Errors support stable `errors.Is` categories and `errors.As` detail. Exact
Codex status/error facts are retained. SDK errors do not invent
provider-neutral classifications.

## Deprecated Handwritten Surface

The following current APIs are migration-only and are not part of the ideal
surface:

- Partial `StartThreadRequest`, `ResumeThreadRequest`, `ForkThreadRequest`, and
  `ThreadClientOptions` defaults.
- Copied `ReasoningEffort`, `ApprovalPolicy`, and `ApprovalsReviewer` enums.
- Copied `InputItem`, `ThreadItem`, and `Usage` models.
- `ThreadEvent` and its projection structs.
- `Bool`, `Text`, and `TextAndFiles` syntax/conversion helpers.

Prefer adding the exact-param core in v0.2 and deprecating published v0.1
surface. Remove retained compatibility declarations no earlier than v0.3.
Before removal, `TextAndFiles` must stop silently discarding blank paths.

## Implementation Sequence

1. Add generator/capability coverage tests distinguishing generated, decoded,
   observable, and projected.
2. Implement exact-param lifecycle composition and exact result snapshots.
3. Implement exact notification stream and full serial handler delivery.
4. Make diagnostics and partial results have real producers.
5. Add typed Codex errors and exhaustive server response mapping.
6. Add migration adapters/deprecations and update docs/release metadata.
7. Add a handwritten export allowlist based on canonical `go/types`
   signatures, excluding generated and internal generator code.

## Required Tests

- Exact start/resume/turn param passthrough and owned ThreadID rejection.
- Exact start/resume/turn response preservation, including partial failures.
- With a non-failing handler and no backpressure failure, every generated
  notification variant reaches the configured full handler.
- Per-run exact notification ordering and immutable result snapshots.
- Ingestion records run evidence before callback enqueue.
- Normal close rejects new work, completes active streams with partial results,
  drains accepted callbacks, and waits for a temporarily blocked handler.
- Handler-returned error, panic, queue overflow, and protocol/transport failure
  cancel all streams with one preserved first cause and do not drain callbacks.
- Handler context cancellation and the documented prohibition on calling
  `Close` from inside the handler.
- Every generated server request kind maps to exactly one response type or a
  documented fail-closed unsupported path.
- `errors.Is`/`errors.As` for protocol, turn, handler, and lifecycle failures.
- Generated method registry, protocol types, SDK surface, and coverage matrix
  reproducibility.
- Handwritten export allowlist and no duplicate generated enums/models.

its permanent `go.mod` to this target API until a real SDK RC or stable tag is

not release evidence.
