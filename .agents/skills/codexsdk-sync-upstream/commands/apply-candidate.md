# Command: apply-candidate

Use when:
- Mechanically applying an already reviewed candidate into the checked-in baseline.

Inputs:
- Candidate schema directory, reports directory, upstream `common.rs`, target ref, target ref kind, and target commit SHA.

Read:
- Top-level skill contract and invariants in `../SKILL.md`.
- `../references/local-sync.md`, "Update Checked-In Baseline" and "Regenerate And Reconcile SDK".

Allowed side effects:
- May update checked-in schemas, baseline metadata, checked-in drift reports, manifest, coverage, generated `protocolv2` Go, and SDK surface through existing scripts.
- May write a local mechanical apply summary.

Forbidden side effects:
- Do not perform agent judgment beyond verifying input provenance matches the resolved target.
- Do not commit, push, tag, create/edit issues, create PRs, request merges, or change branches.

Procedure:
- Verify candidate schema, reports, and response mapping inputs exist and match target provenance.
- Run `scripts/codexsdk_apply_sync_candidate.py`.
- Capture the JSON apply result and `git diff --name-status` when useful for the next command.

Success means:
- Checked-in candidate changes have been mechanically applied and regenerated by canonical scripts.

Validation:
- Apply result JSON parses.
- Changed files match the categories allowed by this command.
- Candidate provenance still matches target ref and commit.

Final output:
- Apply summary path or key findings, diff name-status path or summary, and whether follow-up repair is needed.

Stop rules:
- Stop before mutation if candidate provenance is missing or mismatched.
- Stop if the canonical apply script fails.
- Stop if the script wants to touch files outside the allowed sync surface.
